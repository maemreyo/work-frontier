# Items 20-27 Control Plane Design

## Scope

This batch repairs duplicate SQLAlchemy table registration and delivers the next eight plan items: exclusive writer cutover, gated proposals, WorkLease and Attention lifecycles, break-glass and retention, FastAPI processes, API-only CLI, Control Room onboarding, and the Builder decision workspace.

## Boundaries

- Domain modules own deterministic state transitions and reject stale or unsafe inputs.
- Interface handlers only validate transport data and invoke an injected service contract.
- Persistence is tenant/workspace scoped, RLS protected, and append-only where history is authoritative.
- CLI commands call HTTP endpoints only.
- React consumes generated-shape API types and never computes readiness or ranking.

## Data flow

External projection changes require one exclusive writer lease, exact local/source revision fences, and semantic shadow parity. Authoritative mutations enter an immutable proposal, approval, and disposition flow, then produce a new derived decision. Claims reference a current DecisionRecord and preserve underlying readiness while changing actor-specific claimability. Break-glass creates a short-lived, scoped, audited grant and mandatory review. Retention produces non-PII proof records.

## Processes and UI

The web, worker, and scheduler processes share one application service contract. The Control Room stays in draft onboarding until installation, profile validation, and reconciliation complete. Builder renders API rank one first, exposes decision type, authority, freshness, rationale, blocked roots, and explicit claim conflict handling.

## Failure handling

Cross-scope and malformed identifiers use typed non-leaking responses. Missing writer ownership, stale revision, self-approval, stale DecisionRecords, weak safety overrides, third daily break-glass invocation, and duplicate claims fail closed without partial mutation.
